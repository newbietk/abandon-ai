# synthesize

综合归纳与深度报告生成。三阶段流水线：**Collect**（读取上游报告）→ **Synthesize**（AI 驱动深度交叉分析）→ **Persist**（保存综合报告，保留所有过程数据）。

## 前置条件

本 skill 在工作流中处于最后阶段，依赖以下上游节点全部通过审核：

- `academic_research` — 学术深度研究报告 (`academic_out/academic_report.md`)
- `industry_research` — 行业深度研究报告 (`industry_out/industry_report.md`)
- `practical_research` — 工程实践深度分析报告 (`practical_out/practical_report.md`)

## 完整工作流（6 步）

### Phase A — Collect

#### Step 1: 确认上游输出

验证三份报告文件存在且可读：

```bash
ls -la academic_out/academic_report.md industry_out/industry_report.md practical_out/practical_report.md
```

如果任何文件缺失，中止并报告缺失的报告。

#### Step 2: 深度阅读三份报告

使用 Read 工具完整读取三份报告，提取：
- 各报告的核心发现（含具体证据）
- 方法论/技术架构细节
- 量化数据（benchmark 数字、市场数据、性能指标）
- 关键来源和引用

**阅读时关注点：**
- 哪些发现是三份报告共同指向的？
- 哪些发现之间存在矛盾或张力？
- 哪些重要角度的覆盖存在空白？
- 哪些技术细节可以在交叉分析中产生新洞察？

### Phase B — Synthesize

#### Step 3: 执行深度交叉分析

在撰写综合报告前，先完成以下交叉分析。**这是本 skill 的核心价值所在——不是三份报告的拼接，而是通过交叉分析产生新的洞察。**

**3.1 证据三角验证 (Evidence Triangulation)**

对每个声称的"重大发现"执行三角验证：
- 学术维度是否有理论支撑？
- 行业维度是否有实际数据验证？
- 实践维度是否有工程证据？

输出：每条洞察标注其"验证强度"（三重验证 / 二重验证 / 单维度证据 / 推测）。

```
示例：
洞察：AI 编程工具可提升开发效率 2-5 倍
├── 学术证据：系统综述覆盖 20+ 项研究，一致报告显著效率提升
├── 行业证据：PG&E 年省 527,000 小时员工时间，Lumen 研究时间 4h→15min
├── 实践证据：Cursor Agent 模式实现自动编码→调试→提交全流程
└── 验证强度：★★★ 三重验证（高置信度）
```

**3.2 矛盾与张力分析 (Divergence Analysis)**

识别三个维度之间的矛盾，**必须分析矛盾的根本原因**，而非仅罗列差异：

| 矛盾类型 | 分析框架 |
|---------|---------|
| 时间滞后 | 学术研究通常滞后行业实践 18-24 个月；行业报告可能超前实际落地 6-12 个月 |
| 选择性偏差 | 行业案例倾向于呈现成功案例（幸存者偏差）；学术论文倾向于报告显著结果（发表偏倚） |
| 测量口径差异 | 行业关注商业 ROI，学术关注统计显著性，实践关注开发者体验——三者不一定正相关 |
| 利益冲突 | 厂商报告可能高估效果（推广动机）；独立学术研究可能低估实用性（缺乏工程背景） |

**3.3 缺口分析 (Gap Analysis)**

系统地识别三个维度共同的覆盖空白：
- 哪些关键技术问题未被任何维度覆盖？
- 哪些重要的"失败案例"或"负面结果"缺失？
- 哪些地域/语言/文化视角缺失？
- 哪些时间维度（长期效应）的数据空白？

**3.4 涌现洞察 (Emergent Insights)**

通过交叉对照产生仅凭单一维度无法得出的新洞察：
- 将学术理论映射到工程实践，发现新的应用可能性
- 将行业趋势反向推导出需要学术突破的技术瓶颈
- 将实践中的踩坑经验提炼为可被学术研究验证的假设

#### Step 4: 撰写综合技术报告

创建 `result/` 目录并撰写 `result/{topic_slug}_research_report.md`：

```bash
mkdir -p result
```

## 质量标准

### 综合报告 Checklist

- [ ] 执行摘要精炼（250-350 字），独立可读，含量化数据
- [ ] 学术/行业/实践三个章节均有实质性提炼和重组（非简单复制粘贴大段内容）
- [ ] 交叉洞察包含: 证据三角验证 ≥ 4 条 + 矛盾分析 ≥ 2 条 + 知识空白 ≥ 2 条 + 涌现洞察 ≥ 1 条
- [ ] 每条矛盾标注根本原因分析，而非仅"观点不同"
- [ ] 综合建议覆盖: 研究员 / 技术决策者 / 工程师 三个角色，每条具体可执行
- [ ] 参考来源汇总分类清晰，包含来自三份原始报告的关键来源及深度层级
- [ ] 报告语言与 `inputs.language` 一致（zh 输出中文，en 输出英文）
- [ ] 文件中无占位符 `{xxx}` 残留
- [ ] 所有过程数据保留（不删除上游 raw_r*.json 和 deduped.json）

### 常见问题避免

| 问题 | 正确做法 |
|------|---------|
| 纯拼接三份报告 | 提炼、重组、交叉对照，写出新洞察 |
| 交叉洞察空洞，仅写"三者一致认为..." | 具体引用三个维度中的证据，标注验证强度 |
| 矛盾分析只罗列差异不解释原因 | 分析根本原因: 时间滞后/选择性偏差/测量口径差异/利益冲突 |
| 没有涌现洞察 | 通过交叉映射（理论↔实践）产生独立视角的新发现 |
| 建议泛泛而谈 | 面向具体角色给出可执行的建议，附技术依据和时间估计 |
| 忽略不确定性和空白 | 明确标注知识空白、单维度证据（低置信度）和推测性内容 |
| 来源丢失 | 保留三份原始报告的关键来源链接和深度层级标记 |
| 过度自信 | 对仅有单维度支撑的发现标注"待验证"，区分"结论"和"推测" |

## topic_slug 生成规则

将研究主题转换为合法的文件名 slug：

```python
import re

def topic_to_slug(topic: str) -> str:
    """将中文/英文主题转为文件名 slug"""
    # 移除特殊字符，保留中英文、数字、连字符
    slug = re.sub(r'[^\w\s-]', '', topic)
    # 空白替换为连字符
    slug = re.sub(r'[\s_]+', '-', slug)
    # 去除首尾连字符，转小写
    slug = slug.strip('-').lower()
    # 限制长度
    return slug[:80]
```

示例: `"大语言模型在代码审查中的应用"` → `"大语言模型在代码审查中的应用"` (中文保留) 或 `"llm-code-review-application"` (英文 slug)
